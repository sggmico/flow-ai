# GitHub 博客文章发布规范

## 1. 核心定位

- **目标**: 在技术社区建立专业严谨形象，文章应作为"经得起时间考验的技术手册"
- **受众**: 具有技术背景的开发者、学习者，寻求深度准确的解决方案和技术原理
- **消费习惯**: 深度阅读、动手实践、验证代码，对外部链接和参考资料有较高需求

## 2. Front-matter 规范

必需字段:

```yaml
---
title: 技术准确的标题，面向搜索
date: YYYY-MM-DD
tags: [tag1, tag2, tag3]
description: 50-100 字摘要，说明文章核心价值
---
```
**标题风格**: 技术, 准确, 面向搜索. 标题应清晰地反映文章的核心内容.
**标题示例**: `PWA 渐进式 Web 应用工程指南：从判断到最小闭环`

## 3. 内容风格

### 3.1 语言要求

- **严谨**: 技术表述准确，避免模糊用语
- **深入**: 不仅讲"是什么"，更要讲"为什么"和"如何做"
- **逻辑清晰**: 章节递进有序，论证完整

### 3.2 代码规范

```javascript
// ✅ 正确示例：关键行附有设计意图注释
const cache = new Map();  // 使用 Map 而非 Object，支持任意类型 key

async function fetchWithCache(url) {
  // 缓存命中时直接返回，避免重复网络请求
  if (cache.has(url)) {
    return cache.get(url);
  }

  const response = await fetch(url);
  const data = await response.json();

  // 缓存结果，TTL 由调用方控制
  cache.set(url, data);
  return data;
}
```

要点:
- 代码块必须标注语言类型
- 关键代码行必须附注释，解释**设计意图**而非代码功能
- 复杂逻辑用 Mermaid 图表辅助

### 3.3 可信度建设

- 引用数据或观点时，**必须提供来源链接**
- 技术声明需有官方文档支撑
- 避免主观臆断，用事实和代码说话

## 4. 文章结构

```
# 标题

## 摘要
1-2 段，说明文章价值和读者收益

## 正文章节
- 章节层级不超过 3 级
- 每节聚焦一个主题
- 适时插入代码示例和图表

## 延伸阅读
- 2-3 个高质量官方文档或权威文章链接
- 格式: [链接文字](URL)

## 行动号召 (CTA)
- 提供可运行的 Demo 项目 GitHub 仓库链接
- 鼓励在 Issue 中提出技术问题
```

## 5. 验证清单

发布前必须检查:

- [ ] Front-matter 完整 (title, date, tags, description)
- [ ] 所有链接有效 (内部、外部、Demo 仓库)
- [ ] 代码块正确渲染，注释清晰
- [ ] 引用数据有来源链接
- [ ] 标题适合搜索引擎索引

## 6. 反面示例

❌ **避免**:
- 标题模糊: "关于 PWA 的一些想法"
- 代码无注释或注释重复代码: `// 设置 cache 为 new Map`
- 无来源声明: "据统计 90% 的用户..."
- 缺少实践指引: 只讲理论不给代码
